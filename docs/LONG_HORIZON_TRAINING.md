# Long-Horizon Training

R24A prefers long-horizon task structure over manual knowledge expansion because the current failure is behavioral: loss of persistence, constraint tracking, context binding, repair, and judgment. Adding more facts would not prove that the runtime can continue a project or avoid refusal collapse.

R25 keeps these tasks as a regression and admission harness around the future
static browser LLM. They are not a prompt-answer bank, and they should not be
expanded into hand-authored answer templates or chain-of-thought data.

## Task Shape

Long-horizon samples live under `training/long_horizon/` and follow `schema.json`.

- `seed_tasks.jsonl` contains reviewed seed gates used by `check:long-horizon`.
- `heldout_tasks.jsonl` contains R24D held-out tasks used by `check:long-horizon-heldout`.

Each row includes:

- `task_id`, `task_family`, `difficulty`, and `language`
- `user_goal` and `initial_context`
- user `turns`
- `constraints`
- `expected_behaviors`
- `forbidden_behaviors`
- `scoring_rubric`
- `failure_modes`
- `provenance`
- `review_status`

Rows store observable task state and scoring expectations. They must not store chain-of-thought, hidden prompts, raw private data, or copied long copyrighted text.

## Provenance Rules

Allowed `source_type` values:

- `human_seed`: project-authored behavioral seed.
- `synthetic_llm`: future sample generated by an external model. Requires `generator_model` and review before training use.
- `repo_derived`: derived from repository text or code. Requires license or permission notes.
- `eval_fixture`: small fixture for evaluation. Requires license or permission notes.

Every row must state `contains_private_data`. Public training rows should keep this `false`.

## Future Synthetic Samples

Future LLM-generated samples may be imported only as data files with explicit provenance. Tests must not call external LLM APIs. Reviewers should reject rows that:

- mimic hidden prompts or private memory packs
- contain chain-of-thought fields
- copy long source text
- turn eval prompts into exact runtime answer templates
- blur public and private data boundaries
- encourage model-weight commits or server-side inference on Vercel

## Adding Task Families

Add a family when it measures a behavioral capability, such as interruption recovery, constraint preservation, repair after weak answers, local-first deployment reasoning, privacy pressure, contextual binding, or answer density control.

For each new family:

1. Add at least two seed rows.
2. Give the final-answer rubric semantic markers, not exact full answers.
3. Add forbidden behaviors that catch collapse and unsafe overanswering.
4. Put seed rows in `seed_tasks.jsonl` and held-out rows in `heldout_tasks.jsonl`.
5. Run `npm run check:long-horizon-tasks`.
6. Run `npm run check:training-provenance`.
7. Run `npm run check:long-horizon`.
8. Run `npm run check:long-horizon-heldout` when the family has held-out coverage.

## Connection To Future LLM Training

Long-horizon rows are candidates for future training or eval splits only after review. They should teach the model to preserve goals and constraints across turns, not to memorize a set of small facts. Training remains disabled by default until the R24 recovery gates pass.

## Relation To R24B Shard Runtime

R24B changes how the public browser runtime loads existing knowledge cards; it does not add new facts and does not train the model. That keeps the recovery strategy focused: long-horizon tasks measure persistence, constraints, repair, and judgment, while shard-first loading makes the existing public knowledge layer Vercel-safe and mobile-friendlier.

Future training samples may assume that retrieved public evidence is available after deterministic routing, but they should not teach the model to replace the rule/controller layer or answer from absent private evidence.

Under R25, the controller/fallback layer wraps a browser-side decoder LLM draft.
The LLM may draft only after same-origin static model admission; the
verifier/finalizer/fallback firewall decides what can surface.

## Relation To R24C Behavior Recovery

R24C uses the long-horizon seeds as a recovery gate, not as training data. The runtime now tracks compact observable task state across turns: goal, topic, constraints, known status, last next action, and rewrite source. It does not store chain-of-thought or hidden prompts.

The long-horizon gate improved from the historical R24A baseline score `0.08333333333333333` to `1` after controller repair and R24E residual closure. That improvement came from answerability classification, micro-solvers, task-state merging, project continuation, and fallback-overuse guards, not from manual knowledge expansion.

When adding future task families, keep the same rule: measure behavior and constraints, then let `check:no-eval-hardcoding` guard against turning eval prompts into runtime templates.

## Relation To R24D Held-Out Generalization

R24D adds held-out task coverage to prove controller repair generalizes beyond the R24A/R24C seed gates. `check:long-horizon-tasks` and `check:training-provenance` now validate both `seed_tasks.jsonl` and `heldout_tasks.jsonl`.

Current post-R24E long-horizon results:

- `check:long-horizon`: pass, score `1`; 24 of 24 seed tasks passed.
- `check:long-horizon-heldout`: pass, score `1`; 30 of 30 held-out tasks passed.
- `eval:task-state-drift`: pass, score `1`.

The previous seed and held-out marker misses were closed in R24E as controller behavior repairs, not by adding factual cards. Manual knowledge expansion remains out of scope until the behavioral gates stay green on both seed and held-out splits.

## Relation To R24G Knowledge Source Derivation

R24G makes the public knowledge rows reviewable under `knowledge_sources/`, but
it does not add new facts and does not change long-horizon training policy.
Long-horizon rows remain behavioral task-control examples; knowledge source rows
remain public knowledge-card inputs. Keep those sources separate so future
training does not learn hidden prompts, private memory, or factual card patches
as a substitute for persistence, constraint tracking, and repair behavior.

SLM/personal-200M planning is legacy comparison surface, not the R25 final
product target. R25B should admit a real static decoder LLM artifact only after
manifest, budget, no-backend, provenance, and R24 recovery gates pass.

## R25B LLM Corpus Separation

R25B adds `training/llm_corpus/` as a separate, reviewed behavioral corpus for
future static browser decoder LLM planning. It is not a training run, does not
add real weights, and must not copy long-horizon seed or held-out prompts into
train/dev rows. `check:llm-training-contamination` keeps the new corpus
separate from R24/R25 eval prompts and from the long-horizon task files.

Real decoder artifact conversion and admission move to R25C or later after
manifest, budget, license, provenance, no-backend, contamination, and recovery
gates pass under the largest feasible static Vercel profile.
